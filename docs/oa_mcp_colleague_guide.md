# OA MCP 同事使用标准流程

本文用于发布给同事使用。OA MCP 的目标是让用户在 Agent 对话里完成 OA 待办查询、单据详情查看和标准审批；也支持 OA 文档搜索、详情查看和附件下载。

Agent 对用户的标准回复话术见 [oa_mcp_user_output_standard.md](oa_mcp_user_output_standard.md)。面向普通同事时，优先使用自然语言话术，不展示 MCP 参数、接口地址、token、fdId 等技术内容。

当前稳定版本是 `0.2.12`，审批范围固定为 `standard-approval-v1`：只支持填写审批意见后同意或驳回，不填写或修改业务表单，不选择下一流向，也不执行转办、沟通、废弃、加签、补签等特殊动作。

Agent 必须按连接器返回的处理分级输出：第 1 级“可在对话中审批”、第 2 级“需要先核对”、第 3 级“请到 OA 处理”。无法判断时按第 3 级处理。

## 一句话安装或升级

把下面这句话发给 Agent：

```text
请安装或升级 OA 助手连接器：https://github.com/wildbyteai/oa-agent-connector.git
```

Agent 应按仓库 README 完成安装或升级，然后向用户询问 OA 地址，生成本机 MCP 配置，刷新工具，并引导用户用自己的 OA 账号登录授权。

升级完成后必须刷新或重启 MCP 工具，不能继续使用内存中的旧连接器进程。最新版已兼容“账号可以审批，但流程节点不允许修改申请正文”的情况。

## 一句话说明

安装并授权 OA MCP 后，可以直接对 Agent 说：

```text
查看我的 OA 待办
```

Agent 会只基于当前登录账号的 OA 权限查询待办。待办查询后必须先查看详情；只有标准审批才能在用户二次确认后执行。

也可以说：

```text
在 OA 里搜索：示例产品
```

Agent 会只基于当前登录账号的 OA 权限返回搜索结果。附件下载只允许下载详情页里显示的附件。

## 使用前提

- 用户有自己的 OA 账号和密码。
- 用户已经在本机安装 Python 3.9 或以上版本。
- 用户使用的 Agent 客户端支持 MCP。
- OA 地址为 `<OA_BASE_URL>`。
- 安装 Agent 会在本机生成一个真实固定状态目录，用于保存登录状态和审批确认状态。

## 第一次安装

从发布包或源码目录安装：

```bash
python3 -m pip install .
```

安装成功后，本机应能执行：

```bash
oa-agent-mcp
```

说明：`oa-agent-mcp` 是 stdio MCP server，直接在终端运行时会等待 MCP 客户端输入。普通用户不需要手动常驻运行它。

本机也应能执行配置生成命令：

```bash
oa-agent-mcp-config --base-url "<OA_BASE_URL>"
```

这个命令会自动生成带本机真实状态目录的 MCP 配置，macOS 和 Windows 都兼容。

## MCP 客户端配置

在 MCP 客户端中添加服务：

安装 Agent 应优先运行：

```bash
oa-agent-mcp-config --base-url "<OA_BASE_URL>"
```

然后把命令输出的 JSON 写入 Agent 客户端的 MCP 配置。

如需手动配置，安装 Agent 需要按当前电脑系统生成真实路径，并写入 `OA_AGENT_STATE_DIR`：

- macOS/Linux：当前用户主目录下的 `.oa-agent-connector`，例如 `/Users/yourname/.oa-agent-connector`
- Windows：当前用户目录下的 `.oa-agent-connector`，例如 `C:\\Users\\yourname\\.oa-agent-connector`

下面配置里的 `<ABSOLUTE_STATE_DIR>` 不能原样保留，必须替换成目标电脑上的真实绝对路径。

```json
{
  "mcpServers": {
    "oa": {
      "command": "oa-agent-mcp",
      "env": {
        "OA_BASE_URL": "<OA_BASE_URL>",
        "OA_AGENT_STATE_DIR": "<ABSOLUTE_STATE_DIR>"
      }
    }
  }
}
```

配置完成后，重启或刷新 Agent 客户端，让客户端加载新的 MCP 工具。

## 是否需要常驻服务

不需要用户单独启动后台常驻服务。

MCP 客户端会按它自己的机制启动 `oa-agent-mcp` 子进程，并通过 stdio 与它通信。有些客户端会在会话期间保持该子进程，有些客户端会按需启动；这由 MCP 客户端管理。

用户只需要完成一次 MCP 配置。后续使用时直接在 Agent 里对话即可。

## 第一次授权

用户第一次说：

```text
查看我的 OA 待办
```

如果还没有授权，Agent 应调用 MCP 工具 `oa_begin_auth` 生成本机授权链接，并提示用户点击链接完成授权。

授权时：

- 用户只在本机授权页面输入 OA 账号和密码，不要把密码发在聊天里。
- 密码不会写入聊天记录、连接器文件或 Cookie 文件。用户勾选“在这台电脑上安全记住”后，密码只保存在 macOS 钥匙串或 Windows 凭据管理器。
- `oa_login` 默认不暴露给普通 Agent，避免 Agent 要求用户在聊天里发送密码。
- 本机授权默认优先使用 HTTPS。企业内网 OA 如果只能使用 HTTP，MCP 会先让用户确认正在登录公司 OA；确认后 Agent 按 MCP 返回的 `nextAction` 继续授权，不需要用户手动改配置或重启。`nextAction` 里的确认令牌只给 Agent 内部使用，不展示给用户。
- HTTPS 跳过证书校验需要管理员显式设置 `OA_AGENT_ALLOW_INSECURE_AUTH=1`，不走普通用户确认流程。
- `OA_AGENT_ALLOW_INSECURE_AUTH=1` 只作为管理员预先批准可信内网 HTTP 或 HTTPS 跳过证书校验的全局例外；普通用户安装流程不需要设置。公司统一部署时可由管理员预置，普通用户就不会看到确认步骤。
- MCP 状态目录只保存登录后的 Cookie、OA 地址、登录账号身份提示和待确认审批状态；自动登录密码保存在系统密码保险箱中。
- cookie 和审批确认状态保存在 MCP 配置里的 `OA_AGENT_STATE_DIR`。
- cookie 文件权限会尽量设置为 `0600`。
- MCP 不会因为一次请求失败或鉴权提示就删除 cookie；会保留原文件并引导用户重新登录。

标准授权提示语：

```text
当前还没有完成 OA 授权。

请点击下面的本机授权链接，在页面里输入 OA 账号和密码。

勾选“在这台电脑上安全记住”后，登录过期会自动恢复；密码不会发到聊天或写入连接器文件。

授权成功后，我会继续帮你查询待办。
```

## 日常使用

授权完成后，用户后续可以直接说：

```text
查看我的 OA 待办
```

Agent 标准执行流程：

1. 调用 `oa_list_todos`。
2. 如果成功，列出当前登录账号有权限看到的待办清单。
3. 如果授权失效，MCP 会先使用系统密码保险箱中的登录信息自动恢复并重试原操作。
4. 只有未保存登录信息或自动恢复失败时，MCP 才返回 `reauthRequired=true`。此时不要询问用户是否重新授权，直接按 `nextAction` 调用 `oa_begin_auth`，把本机授权链接发给用户。
5. 如果 MCP 返回 `transportSecurityRequired=true` 且带有 `nextAction`，先提示用户“请确认你正在登录公司 OA”。用户回复“确认继续登录”后，再按 `nextAction` 继续发起本机授权；不要把确认令牌展示给用户。
6. 如果 MCP 返回 `configurationRequired=true`，先让用户提供 OA 地址并重新生成 MCP 配置。
7. 用户完成本机授权后，再次调用 `oa_list_todos`。

待办清单建议展示字段：

- 序号
- 单据标题
- 单号，如 OA 返回
- 申请人，如 OA 返回
- 创建时间，如 OA 返回
- 当前环节
- 当前处理人

普通用户界面不要主动展示技术编号；排查问题时再提供。

示例：

```text
你当前有 2 条 OA 待办：

1. 固定资产领用申请_示例
   单号：DEMO-ASSET-001
   申请人：示例申请人A
   当前环节：上级审批
   当前处理人：审批岗位(审批人A)

2. 请假申请_示例员工(带薪年假：1天)
   单号：DEMO-LEAVE-001
   申请人：示例申请人B
   当前环节：部门负责人
   当前处理人：审批岗位(审批人A)

处理前需要先看详情。你可以说“看第 1 条详情”。
```

## 查看单据详情

用户可以说：

```text
第一个申请我看一下详情
```

Agent 标准执行流程：

1. 将“第一个”映射到刚才待办清单里的第一条 `fdId`。
2. 调用 `oa_get_detail`。
3. `oa_get_detail` 默认只允许查看当前登录账号待办清单里的单据。
4. 将详情整理成人能读懂的摘要。

详情建议展示字段：

- 单据标题
- 单号
- 申请人
- 申请部门
- 申请时间
- 申请事由或业务字段
- 当前环节
- 当前处理人
- 关键明细行

## 搜索 OA 文档和附件

用户可以说：

```text
在 OA 里搜索：示例产品
```

也可以说：

```text
帮我查一下 OA 里有没有示例产品的出厂报告
```

Agent 标准执行流程：

1. 调用 `oa_search_objects`，按用户原话搜索 OA。普通文档搜索优先用 `scope=knowledge`。
2. 保持默认 `requireDetail=true`，只展示能继续打开详情的结果。
3. 用户说“完全匹配某个产品名”时，优先用 `matchMode=contains`；它会忽略 OA 标题中的空格，并默认按文档去重。
4. 不要为了“标题搜索”额外强制下发 OA 标题字段。MCP 会在本地处理标题过滤，避免 OA 搜索接口返回通用错误。
5. 将 OA 返回结果按序号展示给用户，不主动展示技术编号。
6. 用户说“看第 1 条详情”时，调用 `oa_get_object_detail`。
7. 如果 Agent 客户端支持链接，可把结果里的 `detailUrl` 展示成“打开详情”链接。
8. 用户说“下载第 1 条附件”时，先读取详情页附件列表，再调用 `oa_download_attachment`。
9. 下载完成后告诉用户文件名和保存位置。

搜索结果建议展示字段：

- 序号
- 标题
- 创建人
- 创建时间
- 附件数量
- 是否有可查看详情/附件

示例：

```text
OA 里找到以下结果：

1. 出厂报告-示例产品（11#示例色号）
   创建人：示例用户A
   创建时间：2026-07-01
   可查看详情和附件

2. 质检报告-示例产品
   创建人：示例用户B
   创建时间：2026-06-20
   可查看详情和附件

你可以说“看第 1 条详情”，也可以说“下载第 1 条附件”。
```

重要口径：

- `matchMode=contains` 表示标题去空格后包含完整搜索词，适合普通用户说的“完全匹配某个产品名”。
- `matchMode=exact` 表示标题去空格后必须完全等于搜索词，适合标题非常确定的场景。
- 如果使用 `searchFields=["title"]`，MCP 会做本地标题过滤，不会把 OA 会报错的标题字段下发给 OA。
- 默认只展示可继续查看详情的结果；暂不支持详情的结果不能放进用户可选择列表。
- `detailUrl` 是 OA 原生详情页链接；用户浏览器未登录 OA 时，点击后可能先进入 OA 登录页。
- MCP 默认按文档去重，附件级命中会聚合到同一条文档结果里。
- 如果 Agent 自己再做额外过滤、改排序或隐藏结果，必须告诉用户这是 Agent 的二次处理。
- 附件下载不能接受用户提供的任意下载链接，只能下载详情页里列出的附件。

## 审批同意或驳回

审批必须分两步：先准备，后确认执行。

用户表达审批意图：

```text
驳回。备注测试流程
```

Agent 标准执行流程：

1. 先读取详情和 `approvalHandling`，确认这条单据不需要填写或修改业务表单、不需要选择下一流向，也不需要特殊动作。
2. 如果需要上述操作，输出“请到 OA 处理”和详情链接，不调用审批准备工具。
3. 识别动作：同意或驳回。
4. 识别备注：例如 `测试流程`。
5. 调用 `oa_prepare_approval`，只做准备，不提交 OA。
6. MCP 会校验该 `fdId` 是否仍在当前登录账号的待审批清单中，并确认用户选择的同意或驳回是当前节点可用动作。
7. MCP 返回确认摘要、处理分级和 `confirmationToken`。
8. Agent 将确认摘要发给用户，要求用户明确回复固定确认词。

同意的确认词：

```text
确认审批
```

驳回的确认词：

```text
确认驳回
```

标准确认摘要：

```text
请确认本次 OA 审批操作：

单据：固定资产领用申请_示例
当前环节：上级审批
当前处理人：审批岗位(审批人A)
操作：驳回
备注：测试流程
处理权限：已确认这条单据在你的待办里
处理方式：可在对话中审批
范围：本次只提交驳回和审批意见，不填写表单，也不选择下一流向

确认无误后，请回复：确认驳回
```

用户回复：

```text
确认驳回
```

Agent 才能调用 `oa_confirm_approval` 执行审批。执行前 MCP 会再次查询当前登录账号待办清单；如果单据已经不在待办里，会拒绝执行。

## 批量审批

用户也可以一次选择多条待办，例如：

```text
批量处理第 1、2、3 条：第 1、3 条同意，备注同意；第 2 条驳回，备注请补充资料
```

Agent 必须先逐条查看详情并确认每条都属于标准审批，再调用 `oa_prepare_batch_approval`。单次最多 20 条，可以混合同意和驳回；MCP 会一次读取当前账号待办，再逐条确认所选动作和当前审批任务。只有全部项目都通过检查，才会生成批量确认清单。

MCP 的单条和批量正式审批都不能手工指定下一节点，也不能填写业务表单。业务需要这些操作或其他特殊动作时，应让用户在 OA 原生页面处理。

标准确认摘要：

```text
请确认本次批量 OA 审批：

共 3 条：同意 2 条，驳回 1 条

1. 固定资产领用申请_示例 A
   操作：同意
   备注：同意

2. 费用申请_示例 B
   操作：驳回
   备注：请补充资料

3. 采购申请_示例 C
   操作：同意
   备注：同意

处理权限：3 条都已确认在你的待办里，且当前环节支持所选操作。

说明：将按上面的顺序逐条处理。批量审批不是一次性事务；如果中途失败，前面已完成的项目不会回滚，后续项目不会执行。

确认无误后，请回复：确认批量审批
```

用户明确回复：

```text
确认批量审批
```

Agent 才能调用 `oa_confirm_batch_approval`。如果中途停止，Agent 必须用普通语言分别列出：

- 已完成的项目。
- 当前失败或结果不明确的项目。
- 尚未执行的项目。

不得自动重试整批，也不得复用原确认 token。重新处理前先刷新待办，只把仍需处理的项目组成新批次。

如果同一条审批同时存在于另一个批次或单条确认中，MCP 会阻止第二次提交。Agent 对用户只需说明“这条申请正在处理或刚刚提交，我会重新查询待办确认状态”。批量终态可按同一 token 读取，但读取只返回已保存结果，不会再次提交。

## 安全与权限边界

必须遵守以下边界：

- 必须先登录授权，后续操作只使用该登录 cookie。
- 待办查询只返回 OA 现有接口对当前登录账号可见的数据。
- 查看详情默认要求单据存在于当前登录账号待办清单。
- OA 搜索、文档详情和附件下载只使用当前登录账号可见的数据。
- 附件下载只允许下载详情页中列出的可见附件，不能按任意 URL 下载。
- 审批准备和审批执行都会校验单据是否仍在当前账号待办清单中。
- 待办和详情默认输出“需要先核对”；未确认属于标准审批前不能生成确认清单。
- 只有普通同意或驳回和审批意见可以在对话中处理。填写表单、选择流向和特殊动作必须回 OA 页面。
- 审批权限以 OA 详情页显示的当前处理人和可用动作为准，不要求当前账号能够编辑申请正文。
- 当前审批任务和默认驳回节点都遵循 OA 页面自身设置。
- 审批确认绑定准备时的登录账号、流程、任务、节点和动作；其中任何一项变化都必须重新准备并再次确认。
- MCP 不接受任意审批人、任意 handler、任意原始 URL、任意 `flowParam`。
- MCP 禁止直接执行 `oa_approve execute=true` 或 `oa_reject execute=true`。
- 正式审批只能走 `oa_prepare_approval` -> 用户确认 -> `oa_confirm_approval`。
- 批量审批只能走 `oa_prepare_batch_approval` -> 用户确认 -> `oa_confirm_batch_approval`，单次最多 20 条。
- 用户没有明确回复 `确认审批`、`确认驳回` 或批量固定词 `确认批量审批` 时，不允许执行对应审批。
- 批量审批严格串行，首条失败或结果不明确就停止；前面已完成的项目不回滚，后续项目不执行。
- 同一 workitem 跨单条、跨批次也只能提交一次；不能通过生成新 token 绕过防重复锁。

## 异常处理口径

### MCP 没有安装或没有配置

如果 Agent 客户端根本没有加载 OA MCP，Agent 无法调用 MCP 自己给出引导。需要先按本文完成 MCP 安装和客户端配置。

### 未授权或授权失效

如果 MCP 已安装但未授权，Agent 应展示授权指引并调用 `oa_begin_auth`。如果 Cookie 已失效，MCP 应先自动恢复；只有自动恢复不可用或失败时才重新显示授权链接。

如果 MCP 返回 `reauthRequired=true` 且带有 `nextAction`，Agent 应直接进入重新登录流程；不要再问用户“是否重新授权”。MCP 会在 `nextAction` 里带上已配置的 OA 地址和 session，Agent 只需要调用 `oa_begin_auth` 并把授权链接给用户点击。

如果 MCP 返回 `transportSecurityRequired=true` 且带有 `nextAction`，Agent 应先提示用户“请确认你正在登录公司 OA”。只有用户回复“确认继续登录”后，才能按 `nextAction` 继续调用 `oa_begin_auth`；不要向用户展示确认令牌。

如果 MCP 返回 `configurationRequired=true`，说明还没有可用 OA 地址，必须先让用户提供 OA 地址并重新生成 MCP 配置。

标准提示语：

```text
当前 OA 授权不可用，需要重新登录后才能查询待办。

请点击下面的本机授权链接，在页面里输入 OA 账号和密码。

勾选“在这台电脑上安全记住”后，登录过期会自动恢复；密码不会发到聊天或写入连接器文件。
```

### cookie 处理

- 不要因为一次失败请求就删除 cookie。
- 不要在用户不知情的情况下清理会话文件。
- Cookie 明确失效时先自动恢复；没有可用系统凭据或恢复失败时，再调用 `oa_begin_auth`。
- 用户说“关闭 OA 自动登录”时调用 `oa_disable_auto_login`；只删除系统密码保险箱中的登录信息，保留当前 Cookie。
- 如果系统密码保险箱暂时无法清理，告诉用户“自动登录已关闭，但旧登录信息清理未完成，我会重试”；不要显示技术错误。

### 审批确认 token 过期

`oa_prepare_approval` 和 `oa_prepare_batch_approval` 生成的确认 token 默认 15 分钟有效。

如果用户过了很久才确认，Agent 应重新执行对应的准备工具，重新整理确认摘要，再要求用户确认。

### 审批结果不明确

审批请求只会发送一次。OA 没有返回明确结果时，连接器会刷新一次待办获取核对线索：

- 单据仍在待办：结果无法确认。
- 单据已经离开待办：可能是本次操作生效，也可能是他人处理、撤回或流程变化，仍不能宣称成功。
- 待办查询也失败：结果无法确认。

以上情况都要告知用户“结果暂时无法确认，请勿重复提交”，并请用户先打开 OA 页面查看该单据的流程记录。Agent 不得再次调用同一个确认 token，也不得改走其他接口重发审批。

批量审批中某一条结果不明确时，必须立即停止。前面已明确成功的项目列为“已完成”，当前项目列为“结果不明确”，后续项目列为“未执行”；先让用户到 OA 页面核对当前项目，再重新查询待办，不能继续处理后续项目。

## MCP 工具清单

发布给同事时，普通用户不需要直接理解工具参数；这是 Agent 或实施人员使用的接口清单。

- `oa_setup_guide`：返回配置、授权、查询的分步引导。
- `oa_begin_auth`：启动本机授权页面，返回可点击授权链接。
- `oa_local_auth_status`：查询本机授权页面状态。
- `oa_disable_auto_login`：关闭自动登录，保留当前 Cookie。
- `oa_login`：兼容登录工具，保存 cookie，不保存密码；默认不暴露，只有设置 `OA_AGENT_ENABLE_PASSWORD_LOGIN=1` 时才会出现在工具列表里。
- `oa_auth_status`：检查当前 session 是否仍有效，并尽量返回当前登录身份 `loginAs`。
- `oa_list_todos`：查询当前登录账号待办清单。
- `oa_get_detail`：查看待审批单据详情。
- `oa_get_search_schema`：查看支持的 OA 搜索范围、字段和限制。
- `oa_search_objects`：执行 OA 通用只读搜索，默认只返回可查看详情的结果，支持标题去空格匹配和默认文档去重。
- `oa_get_object_detail`：查看搜索结果详情和附件列表。
- `oa_download_attachment`：下载详情页中可见的附件。
- `oa_batch_search_objects`：批量执行 OA 搜索。
- `oa_prepare_approval`：准备审批动作，生成确认摘要和 token，不提交 OA。
- `oa_confirm_approval`：用户明确确认后执行审批，执行前再次校验权限。
- `oa_prepare_batch_approval`：准备最多 20 条批量审批，全部检查通过后生成确认摘要和 token。
- `oa_confirm_batch_approval`：用户回复 `确认批量审批` 后按顺序处理，首条失败或结果不明确即停止。
- `oa_approve`：dry-run 兼容工具，MCP 禁止直接执行。
- `oa_reject`：dry-run 兼容工具，MCP 禁止直接执行。
