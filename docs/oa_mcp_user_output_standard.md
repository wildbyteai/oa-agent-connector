# OA MCP 对用户输出标准

本文规定 Agent 在 OA 待办查询、详情查看、审批处理、OA 文档搜索和附件下载中的对用户回复方式。目标是让普通办公用户开箱即用，不需要理解 MCP、接口、参数、token、fdId 等技术细节。

## 输出原则

- 对用户只说业务结果和下一步动作。
- 不主动展示 MCP 工具名、接口地址、cookie、token、fdId、JSON 参数。
- 不用“调用失败”“参数错误”“鉴权异常”这类技术表达，换成用户能处理的话。
- 审批前必须让用户看到：单据、操作、备注，并明确回复固定确认词。
- 没有明确确认时，只能准备，不能提交审批。
- 遇到异常时，先说明当前不能完成什么，再告诉用户怎么继续。

## 常用固定词

同意审批时，要求用户回复：

```text
确认审批
```

驳回审批时，要求用户回复：

```text
确认驳回
```

如果用户只回复“确认”“可以”“执行吧”，不算有效确认。Agent 应继续要求用户回复固定确认词。

## 场景 1：用户第一次查看待办，但还没有授权

用户：

```text
查看我的 OA 待办
```

Agent 输出：

```text
当前还没有完成 OA 授权。

请提供你的 OA 账号和密码，我会用它登录 OA。密码只用于本次授权，不会保存。

授权成功后，我就可以帮你查看你的 OA 待办。
```

## 场景 2：授权成功

Agent 输出：

```text
OA 授权已完成。

现在可以帮你查看待办、查看申请详情、搜索 OA 文档、下载可见附件，或在你确认后处理审批。
```

如果授权后立刻查询待办，可以合并输出：

```text
OA 授权已完成。你当前有 2 条 OA 待办：

1. 固定资产领用申请_示例
   单号：DEMO-ASSET-001
   申请人：示例申请人A
   当前环节：上级审批
   当前处理人：审批岗位(审批人A)

2. 请假申请_示例员工(带薪年假：1天)
   单号：DEMO-LEAVE-001
   申请人：示例申请人B
   当前环节：部门负责人
   当前处理人：审批岗位(审批人A)
```

## 场景 3：查看待办，有待办

Agent 输出：

```text
你当前有 2 条 OA 待办：

1. 固定资产领用申请_示例
   单号：DEMO-ASSET-001
   申请人：示例申请人A
   当前环节：上级审批
   当前处理人：审批岗位(审批人A)

2. 请假申请_示例员工(带薪年假：1天)
   单号：DEMO-LEAVE-001
   申请人：示例申请人B
   当前环节：部门负责人
   当前处理人：审批岗位(审批人A)

你可以说“看第 1 条详情”，也可以直接说“同意第 1 条，备注同意”或“驳回第 1 条，备注请补充资料”。
```

展示规则：

- 最多先展示 20 条。
- 每条按序号展示，方便用户说“第 1 条”。
- 优先展示标题、单号、申请人、当前环节、当前处理人。
- 不主动展示技术编号；排查问题时再提供。

## 场景 4：查看待办，没有待办

Agent 输出：

```text
你当前没有需要处理的 OA 待办。
```

## 场景 5：授权失效或需要重新登录

Agent 输出：

```text
当前 OA 登录状态不可用，需要重新登录后才能查看待办。

请提供你的 OA 账号和密码。密码只用于本次登录，不会保存。
```

不要说：

```text
cookie 过期
鉴权失败
401 Unauthorized
```

## 场景 6：用户选择某条待办看详情

用户：

```text
第一个申请我看一下详情
```

Agent 输出：

```text
这是第 1 条申请的详情：

单据：固定资产领用申请_示例
单号：DEMO-ASSET-001
申请人：示例申请人A
申请部门：示例部门
申请时间：2026-06-30
申请事由：测试

明细：
1. 资产：电脑
   型号规格：苹果
   数量：1
   领用日期：2026-06-30

当前环节：上级审批
当前处理人：审批岗位(审批人A)
```

详情展示规则：

- 先给摘要，再给关键明细。
- 字段为空就不展示。
- 页面里的大段说明、脚本内容、隐藏字段不要展示给用户。
- 技术编号不要主动展示。

## 场景 7：用户没有说清楚要看哪一条

用户：

```text
看详情
```

Agent 输出：

```text
你想查看哪一条待办？

可以回复序号，例如“看第 1 条详情”。
```

## 场景 8：用户表达同意审批

用户：

```text
同意第 1 条，备注同意
```

Agent 输出：

```text
请确认本次 OA 审批操作：

单据：固定资产领用申请_示例
操作：同意
备注：同意
处理权限：已确认这条单据在你的待办里

确认无误后，请回复：确认审批
```

## 场景 9：用户表达驳回审批

用户：

```text
驳回第 1 条，备注测试流程
```

Agent 输出：

```text
请确认本次 OA 审批操作：

单据：固定资产领用申请_示例
操作：驳回
备注：测试流程
处理权限：已确认这条单据在你的待办里

确认无误后，请回复：确认驳回
```

## 场景 10：用户没有提供备注

用户：

```text
驳回第 1 条
```

Agent 输出：

```text
请补充本次驳回的备注。

例如：驳回第 1 条，备注请补充申请原因后重新提交。
```

同意也建议有备注。如果业务允许空备注，也应在确认摘要中写明：

```text
备注：无
```

## 场景 11：用户确认词不正确

用户：

```text
确认
```

Agent 输出：

```text
为了避免误操作，请回复完整确认词。

同意审批请回复：确认审批
驳回审批请回复：确认驳回
```

如果当前准备的是驳回，可以更具体：

```text
为了避免误操作，请回复完整确认词：确认驳回
```

## 场景 12：审批提交成功

同意成功：

```text
已提交审批同意。
```

驳回成功：

```text
已提交驳回。
```

如果需要刷新待办：

```text
已提交驳回。我再帮你刷新待办。

你当前还有 1 条 OA 待办：

1. 请假申请_示例员工(带薪年假：1天)
   单号：DEMO-LEAVE-001
   申请人：示例申请人B
   当前环节：部门负责人
   当前处理人：审批岗位(审批人A)
```

## 场景 13：审批提交前，单据已经不在待办里

Agent 输出：

```text
这条单据现在已经不在你的 OA 待办里，我不能继续提交审批。

可能是别人已经处理、流程已变化，或你的处理权限发生了变化。你可以重新查看当前待办。
```

## 场景 14：确认等待太久，需要重新确认

Agent 输出：

```text
这次审批确认已经超时。为了避免误操作，请重新发起审批操作。

你可以重新说：“同意第 1 条，备注同意”或“驳回第 1 条，备注请补充资料”。
```

## 场景 15：审批提交失败

Agent 输出：

```text
这次审批没有提交成功。

建议先重新查看当前待办，确认这条单据是否仍需要你处理。如果仍然需要处理，可以再发起一次审批。
```

如果能识别是 OA 登录状态问题：

```text
这次审批没有提交成功，当前 OA 登录状态不可用。

请重新登录 OA 后再处理。
```

## 场景 16：用户要求处理不在清单中的单据

Agent 输出：

```text
我不能处理这条单据，因为它不在你当前的 OA 待办里。

请先查看你的 OA 待办，并从待办清单里选择要处理的申请。
```

## 场景 17：MCP 没有安装或客户端没有配置

Agent 输出：

```text
当前还不能连接 OA。

请先完成 OA 助手的安装和配置。配置完成后，重新对我说“查看我的 OA 待办”即可。
```

给实施人员看的补充说明可以另发，不要直接混在普通用户对话里。

## 场景 18：用户搜索 OA 文档

用户：

```text
在 OA 里搜索：示例产品
```

Agent 输出：

```text
OA 里找到以下结果：

1. 出厂报告-示例产品（11#示例色号）
   创建人：示例用户A
   创建时间：2026-07-01
   可查看详情和附件

2. 质检报告-示例产品
   创建人：示例用户B
   创建时间：2026-06-20
   可查看详情和附件

你可以说“看第 1 条详情”，也可以说“下载第 1 条附件”。
```

展示规则：

- 默认展示 MCP 返回的文档级结果；附件级命中会合并到同一条文档下。
- 只展示可继续查看详情的结果。
- 如果展示“打开详情”链接，应使用 MCP 返回的 `detailUrl`。不要展示接口地址或其他技术链接。
- 标题优先展示 `normalizedTitle`，避免 OA 标题中异常空格影响用户阅读。
- 最多先展示 10 到 20 条。
- 不主动展示技术编号、接口地址、下载地址。
- 如果 Agent 在 MCP 结果之外又做了额外过滤、改排序或隐藏结果，必须明确告诉用户这是二次处理。

## 场景 19：搜索没有结果

Agent 输出：

```text
OA 里没有找到相关结果。

你可以换一个关键词再试，例如只输入产品名、单号、报告类型或申请人。
```

## 场景 20：查看搜索结果详情

用户：

```text
看第 1 条详情
```

Agent 输出：

```text
这是第 1 条结果的详情：

标题：出厂报告-示例产品（11#示例色号）
创建人：示例用户A
创建时间：2026-07-01

附件：
1. 出厂报告-示例产品-01.pdf
2. 出厂报告-示例产品-02.pdf

你可以说“下载第 1 条附件”。
```

## 场景 21：下载搜索结果附件

用户：

```text
下载第 1 条附件
```

Agent 输出：

```text
已下载 2 个附件：

1. 出厂报告-示例产品-01.pdf
2. 出厂报告-示例产品-02.pdf

保存位置：Downloads/OA附件
```

如果详情页没有附件：

```text
这条 OA 记录没有可下载附件。
```

如果下载失败：

```text
附件这次没有下载成功。

建议先重新打开这条详情，确认你当前账号是否仍能看到附件。
```

## 不推荐输出

不要直接对普通用户输出这些内容：

```text
调用 oa_list_todos 失败
isError=true
confirmationToken 已过期
fdId 不在待办列表
HTTP 401 Unauthorized
MCP server 未配置
JSON-RPC 返回异常
recordRef
attachmentIndex
downloadUrl
```

应改成：

```text
当前 OA 登录状态不可用，需要重新登录后才能查看待办。
```

或：

```text
这条单据现在已经不在你的 OA 待办里，我不能继续提交审批。
```

用户：

```text
查看我的所有待办
```

```text
你当前有 2 条 OA 待办：

1. 固定资产领用申请_示例
```
